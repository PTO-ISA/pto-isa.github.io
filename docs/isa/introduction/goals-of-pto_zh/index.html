PTO：设计目标¶

PTO 在 Ascend 软件栈中的目标如下。它们与什么是 PTO 虚拟 ISA中的叙述性介绍和范围与边界中的规范性边界说明互相配合。

核心目标¶

1. 跨代稳定¶

在多代 Ascend NPU 之间保持一套稳定的指令语言。硬件会变，片上存储布局和微架构细节也会变，但底层软件需要一层不用每代都重学一次的公共契约。

例子：针对 PTO ISA 1.0 编写的 TMATMUL kernel，应该能在 A2/A3 和 A5 backend 上正确编译和运行，而不需要改动 PTO 层代码。

2. 性能透明¶

PTO 不打算把机器细节完全藏在一个通用计算 API 后面。Tile 形状、数据搬运、同步、向量微指令和标量控制之所以仍然可见，是因为这些细节往往直接决定 kernel 是“只是正确”还是“真正高效”。

例子：tile-first 模型显式暴露 valid region、layout 和 location intent，让编译器或作者能够避免多余的拷贝与 padding。

3. 为多前端提供统一目标¶

为 C、C++、Python 以及 TileLang、PyPTO 这类 tile 系统和代码生成器提供一个与具体机器代际无关的共同目标。它们应当学一套 PTO 契约，而不是为每代 NPU 重新适配一套低层规则。

4. 提供可分发格式¶

通过 PTOBC 为应用和中间件提供缓存、打包和传输 PTO 程序的形式，而不是过早坍缩成某一种目标相关二进制。

5. 为代码生成器提供共同 ISA¶

把 legalization、变换、特化和验证尽量放在公共 PTO 层完成，再映射到具体硬件代际。这样优化器、翻译器和验证器可以共享同一层语义。

6. 让内核作者可读、可写¶

PTO 不只是编译器的内部输出。它也必须足够明确、足够可读，让人能够手写性能内核、架构测试和低层库代码。

7. 原生支持可扩展并行¶

PTO 从一开始就把并行执行、显式同步和机器可见的数据搬运纳入模型，而不是等系统长大后再补一层“并行扩展”。

非目标¶

PTO 明确不打算解决以下问题：

  • 它不是高层 DSL。 PTO 是低层 ISA，不是张量表达式语言。高层创作应使用 TileLang、PyPTO 或其他 DSL；PTO 是编译目标，而不是源语言。
  • 它不定义硬件专属微操作。 PTO 只暴露架构可见行为。TADD 在 A5 上最终拆成 1 条还是 16 条微操作，是 backend 实现细节，不属于 ISA 手册。
  • 它不取代 backend 编译器。 PTO 程序仍需经过 backend（例如 bisheng）降到目标相关二进制。PTO 定义 PTO 层契约；backend 负责优化和指令选择。
  • 它不承诺跨 profile 位级完全一致。 CPU 仿真、A2/A3 和 A5 可能因为微架构不同而产生轻微浮点差异。PTO 保证的是各 profile 语义内的数值正确性，而不是跨 profile 的 bit-exact 身份。

目标冲突时的优先级¶

当这些目标互相拉扯时，PTO 按下面的顺序取舍：

  1. 正确性：非法程序必须被拒绝，合法程序必须给出正确结果。
  2. 可移植性：同一 PTO 程序必须能在所有 profile 上正确运行，并遵守各自的 profile 约束。
  3. 性能：PTO 应暴露足够多的细节以支持手工优化，但不把具体微优化策略写死进 ISA。
